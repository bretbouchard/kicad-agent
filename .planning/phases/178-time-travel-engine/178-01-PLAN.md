Create TimeTravelEngine per ARCHITECTURE.md Pattern 3:

1. **TimeTravelEngine.swift**:
   - `struct TimeTravelEngine { let context: ModelContext; let daemonClient: DaemonMCPClient }`
   - `func replayEvents(upTo eventIndex: Int, from conversation: Conversation) async throws -> Conversation`: 
     - Fetch all Decision/ValueChange events up to eventIndex with predicate `conversationId == conversation.id AND eventIndex <= target`.
     - Sort by timestamp ascending.
     - Deep copy conversation (Conversation.copy() method).
     - Apply events in order: Decision events update conversation.decisions, ValueChange events apply to a mutable state dictionary.
     - Return replayed conversation state.
   - `func createSnapshot(at eventIndex: Int, for conversation: Conversation) async throws -> ProjectSnapshot`:
     - Call replayEvents() to get conversation state at eventIndex.
     - Call PipelineOrchestrator (from Phase 183) to regenerate KiCad files from replayed state.
     - Read schematic/pcb Data from files.
     - Compute stateHash (SHA256 of schematicData + pcbData).
     - Insert ProjectSnapshot with isAutoSnapshot flag (false for manual, true for auto on decisions per MEM-07).
     - Return snapshot.

**Snapshot immutability (TT-07)**: ProjectSnapshot.schematicData and pcbData are `@Attribute` with no public setters. SwiftData generates private setters only. Add computed property `var isImmutable: Bool { true }`. Test verifies setter fails compile.

**Linked to originating message (TT-02)**: Decision.linkedMessageId field populated when Decision created. Query: `fetch Message where id == decision.linkedMessageId`.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Snapshot restore | User can revert project state — must validate restored files |
| Diff computation | Line-by-line comparison may be expensive on large files |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-178-01 | T (Tampering) | Snapshot stateHash | mitigate | SHA256 computed from schematicData + pcbData. Reject on hash mismatch. |
| T-178-02 | I (Information disclosure) | Diff rendering | accept | Diff shows KiCad file changes (no secrets). Acceptable for v1. |
| T-178-03 | D (Denial of Service) | Diff on 1M+ line files | mitigate | Chunked reads, timeout 10s. Pitfall 8 prevention. |
| T-178-04 | E (Elevation) | Restore bypasses gates | mitigate | Validation gates (ERC/DRC) run post-restore. Fail = Obdurate escalation. |
| T-178-05 | R (Repudiation) | Restoration event log | mitigate | Restoration Decision auto-logged with timestamp + actor. |

## Adversarial Scenario

**Name:** Snapshot restore bypasses validation gates, introduces corrupt KiCad files

**Attack:** User restores to snapshot with malicious schematicData (modified externally). TimeTravelEngine writes corrupted .kicad_sch without ERC validation. KiCad GUI crashes on open.

**Mitigation:** Post-restore validation gates (ERC/DRC) mandatory. If validation fails, restoration rejected, error surfaced to user. Test `testRestoreToCorruptSnapshotFailsValidation` verifies ERC runs, throws error on corrupt files.

**Detection:** Unit test mocks corrupt schematicData, expects TimeTravelEngine.restore to throw ValidationError.
</threat_model>
