<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| CI runner → source code | CI runner executes SwiftLint and XCUITest on untrusted PR code |
| XCUITest → macOS accessibility APIs | Tests interact with system VoiceOver/Reduce Motion preferences |
| SwiftLint → .swiftlint.yml | SwiftLint loads custom rules file (inject rule could bypass) |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-201-01 | S | SwiftLint rules file | mitigate | Use pinned SwiftLint version in CI, verify .swiftlint.yml checksum in workflow |
| T-201-02 | T | XCUITest preference changes | mitigate | Tests reset preferences in tearDown, isolate test runs to prevent system state leakage |
| T-201-03 | I | Malicious PR injects .swiftlint.yml | accept | PR review required for .swiftlint.yml changes (repo maintainers only) |
| T-201-04 | D | VoiceOver test flakiness | mitigate | Use XCUIAccessibility utilities (deterministic), retry tests 3x on failure |
| T-201-05 | E | CI evasion via disabled workflow | mitigate | Require branch protection for main, workflow files owned by admins |

## Adversarial Scenario

A malicious contributor submits a PR that modifies .swiftlint.yml to disable the a11y rules, then introduces code that removes accessibilityLabels from Buttons. Without mitigation, the PR would pass CI and ship broken a11y.

Mitigation: 
1. .swiftlint.yml is protected (branch protection requires admin approval for changes)
2. CI workflow verifies .swiftlint.yml checksum before running SwiftLint
3. XCUITest suite still fails (keyboard tests fail on unlabeled Buttons)
4. Defense in depth: SwiftLint + XCUITest + PR review
</threat_model>
