  <name>Task 1: Integrate CKShare and CloudKit collaboration</name>
  <files>KiCadAgent/Collaboration/CKShareManager.swift</files>
  <action>
Create CKShareManager following ARCHITECTURE.md collaboration patterns:

**CKShare setup:**
- CKShare(url: URL) wraps .kicadagent bundle in iCloud Drive
- CKShareMetadata: title, type, icon
- participantPermission: default = .readOnly (view permission per PROJECT.md decision)
- Set .userRole = .owner (current user is project owner)

**Share Sheet integration:**
- shareProject(_ project: Project) async throws → UICloudSharingController
- Presents native macOS Share Sheet (Mail, Messages, copy link)
- Universal link format: kicadagent://invite?shareToken=...
- Share URL includes CKShare URL + access token

**Collaborator management:**
- listCollaborators() async throws → [CKShare.Participant]
- Fetches from CKShare.participants array
- Returns participant info: name, email, permission, status (.accepted/.pending/.declined)

**Permission model (COLLAB-03):**
- enum ProjectPermission: view, edit, fork
- Maps to CKShare.ParticipantPermission:
  - view → .readOnly
  - edit → .readWrite
  - fork → .readOnly (fork creates new project, no edit on original)
- Default permission = view (per PROJECT.md decision)

**CKShare lifecycle:**
- On share: create CKShare, upload to CloudKit, generate share URL
- On collaborator join: CKShare participant added to .participants array
- On permission change: update CKShare.participant[x].permission
- On revoke: remove participant from .participants array
</action>

<task type="auto">
  <name>Task 2: Implement permission management and revocation</name>
  <files>KiCadAgent/Collaboration/PermissionManager.swift</files>
  <action>
Create PermissionManager for ACL enforcement:

**Permission upgrade/downgrade:**
- updatePermission(for: CKShare.Participant, to: ProjectPermission) async throws
- Validates user is .owner (only owner can change permissions)
- Updates CKShare.participant[x].permission
- CloudKit sync propagates to all devices

**Permission checks (Pitfall 10 prevention):**
- canEdit(project: Project, user: User) → Bool
- Check CKShare.userRole: .owner/.admin → true
- Check CKShare.participantPermission: .readWrite → true
- Client-side enforcement BEFORE server-side rejection
- Disable edit UI if !canEdit (per Pitfall 10)

**Access revocation (COLLAB-09):**
- revokeAccess(for: CKShare.Participant) async throws
- Validates user is .owner
- Removes participant from CKShare.participants array
- CloudKit sync triggers invalidation on collaborator's device

**Cache invalidation (COLLAB-09):**
- On revocation: CKShare triggers .participantDidChange callback
- Collaborator device receives CKShare metadata update
- Detects participantStatus = .removed or no longer in .participants
- Clear local SwiftData cache for that project
- Show "Access revoked" state, purge local .kicadagent bundle copy
- Force app reload to apply new ACL

**Apple ID requirement (COLLAB-02):**
- On invite accept without Apple ID: show "Sign in with Apple ID" prompt
- Uses AuthenticationServices framework: ASAuthorizationAppleIDProvider
- One-tap sign-in, returns Apple ID credential
- Store Apple ID in Keychain (device-local, iCloud sync opt-in)
- After sign-in: retry invite accept

**Expired invite handling (COLLAB-04):**
- Detect CKShare.expirationDate < Date.now
- Show "Invite expired. Request new invite from owner." flow
- "Request new invite" button → emails owner with renewal request
- Owner receives email, clicks link to generate new CKShare URL

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Share Link → Collab Device | Untrusted invite URLs, expired tokens |
| Permission Checks → UI | Client-side ACL must match server-side CKShare |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-188-01 | S | Invite URLs | mitigate | Validate CKShare URL format, reject malformed links, HTTPS only |
| T-188-02 | T | Permission checks | mitigate | Client-side check before operation (Pitfall 10), disable edit UI for .readOnly |
| T-188-03 | I | Permission elevation | mitigate | Only owner can change permissions, validate .userRole = .owner before update |
| T-188-04 | D | Cache invalidation | mitigate | CKShare callback triggers immediate purge, force app reload |
| T-188-05 | E | Share link flooding | accept | CKShare server-side rate limiting, expire links after 7 days |

## Adversarial Scenario

Name: Permission Elevation Attack
Collaborator with view permission crafts malicious request to CKShare server attempting to elevate to .readWrite without owner approval.

Mitigation:
1. Server-side CKShare validates .userRole = .owner before permission change
2. Client-side check: PermissionManager.updatePermission throws if !isOwner
3. Audit log: all permission changes logged with actor, timestamp, old/new value
4. Owner notification: "Permission changed" alert on collaborator upgrade
</threat_model>
