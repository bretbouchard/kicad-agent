  <files>Volta/Collaboration/ProjectBundle.swift, Info.plist</files>
  <action>
Create ProjectBundle manager following Pitfall 6 prevention patterns:

**Bundle structure (FILE-03):**
- .kicadagent bundle is a directory (file package) with extension .kicadagent
- Contents:
  - conversation.jsonl — Append-only message/event journal (JSONL format)
  - decisions.json — All decisions with metadata (JSON array)
  - values.json — Current value state (JSON object)
  - schematic.kicad_sch — Generated schematic (KiCad 10 format)
  - pcb.kicad_pcb — Generated PCB (KiCad 10 format)
  - renders/ — Directory with PNG renders, SVG previews
  - metadata.json — Bundle version, timestamps, checksums

**Document type declaration (Info.plist):**
- CFBundleDocumentTypes array with .kicadagent entry
- CFBundleTypeExtensions: ["kicadagent"]
- CFBundleTypeMIMETypes: ["application/x.volta"]
- LSItemContentTypes: ["com.kicad.agent.bundle"]
- CFBundleTypeRole: Editor
- CFBundleTypeIcon: BundleDocument.icns (document icon)
- NSExportableTypes: UTI declaration for macOS

**Atomic writes (Pitfall 6 prevention):**
- Use NSFileCoordinator for all bundle writes
- Pattern:
  1. Write to temporary location (tmp.kicadagent)
  2. NSFileCoordinator.coordinate(writingItemAt: tempURL)
  3. Atomic replace: NSFileManager.default.moveItem(at: tmpURL, to: bundleURL)
  4. NSFilePresenter trigger: presentItemDidChange()
- Never write individual files inside bundle (causes iCloud Drive partial sync issues)
- Always replace entire bundle atomically

**Bundle reads:**
- NSFileCoordinator.coordinateReadingItemAt: for bundle access
- Read conversation.jsonl line-by-line (lazy load for large journals)
- Parse JSON files (decisions.json, values.json)
- Load KiCad files via kicad-python (existing library)

**Bundle validation:**
- Checksum validation: SHA256 of KiCad files vs metadata.json checksums
- JSON schema validation: conversation.jsonl, decisions.json, values.json
- Missing file detection: if schematic.kicad_sch missing, flag as corrupt

**iCloud Drive sync (FILE-02):**
- Bundle stored in ~/Library/Mobile Documents/com~iCloud~com.kicad.agent/Documents/
- iCloud Drive auto-syncs to user's account (no dev infra)
- NSFilePresenter callbacks: presentItemDidChange(), presentedItemDidMove()
- On iCloud conflict: detect via NSFileVersion, prompt user for resolution

<task type="auto">
  <name>Task 2: Implement corrupt bundle detection and repair</name>
  <files>Volta/Collaboration/BundleRepairManager.swift</files>
  <action>
Create BundleRepairManager for event-sourced regeneration (FILE-04):

**Corruption detection:**
- validateBundle(_ bundleURL: URL) throws → BundleValidationResult
- Checks:
  - All required files exist (conversation.jsonl, decisions.json, values.json, schematic.kicad_sch, pcb.kicad_pcb)
  - JSON parses without errors
  - Checksums match (SHA256 of KiCad files vs metadata.json)
  - conversation.jsonl has valid JSONL format
- If any check fails: bundle is corrupt

**Repair flow (FILE-04):**
- On corrupt bundle detected: show "Bundle corrupted. Repair?" prompt
- Repair = event-sourced regeneration:
  1. Load conversation.jsonl (event journal)
  2. Load decisions.json, values.json (SwiftData state)
  3. Rebuild SwiftData models from events
  4. Regenerate KiCad files via daemon (skidl_to_kicad_sch, skidl_to_kicad_pcb)
  5. Validate regenerated files (ERC/DRC)
  6. Write new bundle atomically via ProjectBundle
- Show progress: "Repairing... Replaying events (N/1000)"
- On success: "Bundle repaired. X decisions recovered."
- On failure: "Repair failed. Error: [error details]. Contact support?"

**Repair UI:**
- Alert: "This project bundle is corrupted."
- Explanation: "Some files may be missing or invalid. We can attempt repair by regenerating from your conversation history."
- "Repair Bundle" button (primary action)
- "Cancel" button (dismisses, opens in read-only mode)
- Progress bar during repair with step indicator
- Success/failure alerts with details

**Prevention:**
- Validate bundle after every write
- Checksum validation on read
- Detect iCloud Drive conflict copies (.conflict files)
- Auto-resolve conflicts: prompt user to choose version (similar to LWW conflict resolution)

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| iCloud Drive → Local Storage | Untrusted bundle downloads, potential corruption |
| Zip Import → Bundle Restoration | Untrusted zip files, malicious content |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-190-01 | S | Zip import | mitigate | Validate zip structure, reject malformed bundles, sanitize file paths (prevent path traversal) |
| T-190-02 | T | Bundle writes | accept | iCloud Drive encrypted in transit, Apple handles security |
| T-190-03 | I | Event replay repair | mitigate | Validate conversation.jsonl schema before replay, limit event count (max 100K), timeout after 30s |
| T-190-04 | D | Bundle corruption | mitigate | Checksum validation, atomic writes prevent partial states, repair available |
| T-190-05 | E | Zip export flood | mitigate | Rate limit exports (max 10 per hour), warn on large files (>100MB) |

## Adversarial Scenario

Name: Malicious Zip Import Attack
Attacker crafts malicious .zip file with path traversal (../../etc/passwd) or corrupted JSON to corrupt KiCad Agent or gain file system access.

Mitigation:
1. Validate zip structure: only allow flat bundle structure (no parent directory references)
2. Sanitize filenames: reject ../, ~/, absolute paths
3. Validate JSON schema: reject malformed conversation.jsonl before SwiftData insert
4. Sandbox: app runs in macOS sandbox, can only write to designated directories
5. Repair availability: if import corrupts bundle, offer repair via event replay
</threat_model>
