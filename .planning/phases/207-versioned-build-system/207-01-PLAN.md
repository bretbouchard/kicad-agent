# Phase 207 Plan 01 — Versioned Build System

<phase_goal>
User can create a versioned build that snapshots source files, records git SHA + board revision, and serializes a manifest with SHA256-hashed artifacts to disk. One plan, five tasks, ten BUILD requirements.
</phase_goal>

<phase_context>
Phase 205 shipped `NativeTitleBlock.rev` (frozen dataclass at `pcb_native_types.py:344`). Phase 206 shipped `drc_vendor`/`list_vendor_drc_profiles` as query ops. Phase 207 builds the `Build` record model, promotes `ManufacturingManifest` with serialization, and adds three query ops (`build_create`, `build_list`, `build_show`) that create side-effect artifacts in `builds/` without touching the `.kicad_pcb` source.

**Key design decisions (from 207-CONTEXT.md + 207-RESEARCH.md):**
1. All three ops are `category: "query"`, `is_readonly: True` — NOT CROSS_FILE_OP_TYPES (IP-4 deviation, CONTEXT.md:87-89). The handler creates `builds/` artifacts; `execute_query` never serializes the target `.kicad_pcb`.
2. Registry count 156 → 159 (+3 ops). `test_readonly_operations_count` asserts an exact frozenset — must add all 3 ops.
3. Handler must re-parse via `NativeParser.parse_pcb(file_path)` for `board_rev` (dual-path issue: query path has `_native_board = None`, same as Phase 205/206).
4. `ManufacturingManifest`/`ManufacturingArtifact` are frozen dataclasses — add explicit `to_json/save/load` (manifest) and `to_dict/from_dict` (artifact). `dataclasses.asdict` is one-way; load must reconstruct tuples explicitly.
5. `build_show` accepts optional `diff_build_id` for diffing — NOT a separate op (keeps registry at +3, not +4). Diff logic lives in `diff_builds()` utility.
6. Manifest serialization uses `atomic_write` (same as Phase 205 `save_board_spec`).
7. Validation is simplified for v1: `build_create` default produces `BuildStatus.DRAFT` (PCB parses check). Full `ManufacturingReadinessGate` requires context (DRC/DFM/exports) that Phase 208's handoff orchestrator provides. DRAFT status is honest (avoids Pitfall 5 false confidence).
8. Single-file on disk: `manifest.json` carries the full Build envelope (build_id, git_sha, status, source_files, build_dir + manifest fields). Success criterion #3 requires "reloading reproduces the build record exactly."
</phase_context>

<must_haves>

<threat_model>
**Attack surface for Phase 207:**

1. **Build directory creation (path traversal via project_dir):** `build_create` resolves `project_dir` from `op.project_dir` (user-controlled string) and creates `builds/v{rev}_{timestamp}/` under it. If `project_dir` contains `../` sequences, an attacker could write the build dir outside the intended project root.
   - **Mitigation:** Reject `project_dir` values containing `..` path segments. The schema field `project_dir` should validate that the resolved path stays within a reasonable bound (or at minimum reject obvious traversal). The handler should use `Path.resolve()` and verify the resulting `build_dir` is within the resolved `project_dir`. Board rev used in directory name comes from `title_block.rev` (also user-controlled via the PCB file) — sanitize it to `[A-Za-z0-9._-]` only before interpolating into the path.

2. **Git SHA capture (subprocess injection via project_dir):** `subprocess.run(["git", "rev-parse", "HEAD"], cwd=str(project_dir))` passes `project_dir` as `cwd`. Since the command is a fixed argument list (NOT a shell string), shell injection is not possible. However, a malicious `project_dir` pointing to an attacker-controlled directory runs `git` in their context.
   - **Mitigation:** Use the same `project_dir` validation as build dir creation. The argument-list form (no `shell=True`) prevents command injection. Wrap in `try/except (subprocess.SubprocessError, OSError)` returning `"unknown"` — never raises.

3. **Manifest JSON serialization (injection via board name/project name):** `ManufacturingManifest` fields (`project_name`, `board_name`) come from the PCB file or project. These flow into `manifest.json` via `json.dumps`. Since `json.dumps` escapes all special characters, JSON injection is not possible. However, if these values are later rendered into a readme (Phase 208), they need HTML escaping then — NOT a Phase 207 concern.
   - **Mitigation:** `json.dumps` handles serialization safely. No additional escaping needed for Phase 207. Board rev used in directory names is sanitized (see #1).

4. **Source file snapshot (copying files outside project dir):** `shutil.copy2` copies `.kicad_pcb`, `.kicad_sch`, `.kicad_pro` with the same stem as the target file. Discovery is stem-based within `project_dir` — it cannot copy arbitrary files outside the project because the candidates are constructed as `project_dir / f"{stem}{ext}"`.
   - **Mitigation:** The stem-based discovery is inherently bounded. Validate that resolved candidate paths are within `project_dir` (defense in depth). Never follow symlinks out of the project dir.
</threat_model>
