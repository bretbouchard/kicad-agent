<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Swift app → MCP call_tool | Untrusted tool invocation crosses here — all op arguments must be validated |
| MCP call_tool → OperationExecutor | Trusted internal boundary — executor validates file operations |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-168-01 | Spoofing | Tool name validation | mitigate | Validate op_type against registry before execution, reject unknown ops |
| T-168-02 | Tampering | Tool arguments | mitigate | Pydantic schema validation on all arguments, reject invalid types/ranges |
| T-168-03 | Repudiation | Tool execution audit | mitigate | Log all tool calls to stderr with timestamp, op_type, arguments (sanitized), and result status |
| T-168-04 | Information disclosure | Schema exposure | accept | Tool schemas are public metadata — no secrets exposed |
| T-168-05 | Denial of Service | Resource exhaustion | mitigate | OperationExecutor has file scope validation, no network ops, bounded execution time |
| T-168-06 | Elevation of Privilege | File write operations | mitigate | Executor validates file paths against base_dir, no directory traversal attacks |

## Adversarial Scenario

**Scenario:** Malicious Swift app sends {"jsonrpc":"2.0","id":"attack","method":"call_tool","params":{"name":"remove_component","arguments":{"target_file":"../../../../etc/passwd","reference":"root"}}}

**Mitigation:**
- Pydantic schema validates target_file is Path type
- OperationExecutor normalizes paths and checks they're within base_dir
- Path traversal attempts are rejected with error: "Path outside project boundary"
- Audit log records attempt with sanitized arguments

**Response:** MCP server returns error {"error":{"code":-32602,"message":"Invalid params","data":"Path outside project boundary"}}. No file operations outside base_dir.
</threat_model>
