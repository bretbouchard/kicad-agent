<!-- ME-110-09: b2sdk v2 atomicity -->
b2sdk v2 has NO native atomic rename for objects in a bucket (unlike S3). B2 has copy-then-delete semantics. For checkpoint overwrites (same key, new content), the pattern is:
1. Upload to a temporary key (e.g., `phase-110/step-100.safetensors.tmp`)
2. Copy temp -> final key (`phase-110/step-100.safetensors`) — B2 copy is server-side, atomic for the destination
3. Delete the temp key
Readers always see either the previous final or the new final, never a partial upload. SHA1 verification on upload catches any in-flight corruption (b2sdk does this automatically for `upload` calls when `file_info` includes SHA1).
