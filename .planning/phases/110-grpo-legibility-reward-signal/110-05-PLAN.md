## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Local Mac → Vast.ai instance | Code + data uploaded to third-party GPU host. B2 credentials live on instance. |
| Vast.ai host → B2 bucket | Checkpoint uploads cross provider boundary. |
| Trained adapter → Local Mac → Production | Adapter weights are untrusted model artifacts until validated. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-110-17 | Denial of Service | Vast.ai host preemption | mitigate | SIGTERM handler flushes checkpoint before kill; B2 upload every 50 steps; resume_from_latest() handles restart. Per vastai-training-lessons.md, hosts WILL go offline — this is expected, not exceptional. |
| T-110-18 | Information Disclosure | B2 credentials on instance | mitigate | Credentials passed via env vars (not committed to repo). Instance disk wiped on deallocation per Vast.ai policy. |
| T-110-19 | Tampering | Checkpoint corruption | mitigate | Atomic local write (tempfile + os.replace). Atomic B2 upload (tempfile + rename). resume_from_latest validates file parses before returning. |
| T-110-20 | Tampering | Trained adapter weights | mitigate | Operator verification (Task 3 checkpoint) confirms adapter loads in known-good inference pipeline. SRS scoring comparison vs baseline catches degenerate adapters. |
| T-110-21 | Repudiation | Training run audit trail | mitigate | TRAINING-RUN-LOG.md captures instance ID, hyperparams, metrics. B2 object metadata captures upload timestamps. |

## Adversarial Scenario

A Vast.ai host fails catastrophically at step 1950/2000 (this literally happened per vastai-training-lessons.md — instance 41735651 went offline at step 840/2000). Without the CheckpointResumer, all training progress is lost. WITH the resumer: the SIGTERM handler catches the preemption signal (Vast.ai sends SIGTERM ~30 seconds before hard kill), flushes the step-1900 checkpoint to B2 atomically, and `resume_from_latest()` on a fresh instance downloads step-1900 and continues. Net loss: ~50 steps instead of 1950. The 30-second SIGTERM window is tight but sufficient for a single LoRA checkpoint (~50 MB at rank 16).
