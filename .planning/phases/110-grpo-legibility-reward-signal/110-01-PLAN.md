<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| CritiqueResult → LegibilityReward | Untrusted JSON from Phase 109 critic (Gemma 4 vision model output). May be malformed, missing keys, or contain out-of-range floats. |
| Config → Cap thresholds | User-supplied thresholds via config.json. Could be negative, NaN, or extremely large. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-110-01 | Tampering | LegibilityReward.score() | mitigate | Validate all 4 factor keys present (KeyError if missing); validate each value in [0.0, 1.0] (ValueError if out of range). Fail fast — no silent defaults that could mask a broken critic. |
| T-110-02 | Tampering | CompactnessCap | mitigate | Guard against div-by-zero (footprint_mm2=0 → use max(footprint_mm2, 1.0)). Negative bbox raises ValueError. |
| T-110-03 | Information Disclosure | AlignmentJitter | accept | Jitter is data augmentation, not security-sensitive. RNG seed must be deterministic for reproducibility (test 11), not for secrecy. |
| T-110-04 | Elevation of Privilege | reward module overall | accept | Pure compute module — no file I/O, no network, no subprocess. Caps are frozen dataclasses. Attack surface minimal. |

## Adversarial Scenario

A malicious or hallucinating Phase 109 critic emits `{"density": 1e9, "clarity": 1e9, ...}` (out-of-range floats) trying to inflate the reward signal and cause the GRPO loop to converge on a degenerate policy. Mitigation: Test 7 in Task 1 explicitly validates factor values are within `[0.0, 1.0]` and raises `ValueError` — the GRPO loop (Plan 04) catches this per-step and treats the sample as reward-zero, preventing the inflation from propagating into the policy gradient.
</threat_model>
