## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-110-13 | Tampering | RewardWeights.__post_init__ | mitigate | Validate weights sum to 1.0 ± 1e-6 (training stability). Reject NaN/inf. |
| T-110-14 | Denial of Service | compute_group_rewards | mitigate | adapter=None path preserves existing behavior (Test 2). No forced migration. |
| T-110-15 | Information Disclosure | reward_decomposition log | accept | Logs only float values, no schematic contents or secrets. |
| T-110-16 | Repudiation | config-driven weights | mitigate | Weights logged at trainer init via logger.info("reward_weights=%s", weights) for audit trail. |

## Adversarial Scenario

A user sets `reward_weights: {correctness: 0.0, completeness: 0.0, legibility: 1.0}` in config.json, intentionally or accidentally. This causes GRPO to optimize purely for legibility, producing models that generate beautiful but electrically broken schematics. Mitigation: (1) Test 8 in Task 1 only checks weights sum to 1.0, doesn't enforce per-term bounds — this is intentional (user may want to ablate a term to 0 for debugging); (2) HOWEVER, the per-term contribution logging (Test 5) makes the configuration choice visible in every training log line, so the misconfiguration is observable, not silent; (3) the D-03 default (0.4/0.4/0.2) is hard-coded in RewardWeights dataclass defaults — config.json overrides must be explicit.
</threat_model>
