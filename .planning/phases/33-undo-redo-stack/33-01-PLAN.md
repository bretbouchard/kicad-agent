The module contains:

1. `UndoEntry` frozen dataclass with fields:
   - `file_path: Path` -- resolved path to the mutated file
   - `pre_content: str` -- content before mutation
   - `post_content: str` -- content after mutation
   - `op_type: str` -- operation type for display/logging
   - `post_mtime: int` -- mtime_ns after commit, for stale snapshot detection (L-05)

2. `UndoStack` class:
   - `__init__(self, max_size: int = 50)` -- raises ValueError if max_size < 1. Creates `_undo: dict[Path, deque[UndoEntry]]`, `_redo: dict[Path, deque[UndoEntry]]`, `_lock = threading.Lock()`, `_max_size = max_size`
   
   **Memory warning (M-07):** The docstring should document: "Each UndoEntry stores two full file content strings. With default max_size=50 and typical KiCad files of 100KB-1MB, memory usage is ~10-50MB per file. For large PCB designs (>5MB), reduce max_size via KICAD_UNDO_MAX_SIZE env var."

   - `push(self, file_path: Path, pre_content: str, post_content: str, op_type: str, post_mtime: int = 0) -> None` -- resolves path, creates UndoEntry (with post_mtime), appends to `_undo[resolved]` deque (creating with maxlen if not present), clears `_redo[resolved]` (pop the key, not just clear)
   
   - `pop_undo(self, file_path: Path) -> Optional[UndoEntry]` -- resolves path, pops from `_undo[resolved]`, appends to `_redo[resolved]` (creating deque if needed), returns entry or None
   
   - `pop_redo(self, file_path: Path) -> Optional[UndoEntry]` -- resolves path, pops from `_redo[resolved]`, appends to `_undo[resolved]`, returns entry or None
   
   - `can_undo(self, file_path: Path) -> bool` -- thread-safe peek
   - `can_redo(self, file_path: Path) -> bool` -- thread-safe peek
   - `clear(self) -> None` -- clears both dicts under lock
   - `max_size` property returning `_max_size`
   
   **H-03 fix -- simplified latest tracking:** Instead of maintaining `_latest_undo`/`_latest_redo` fields (which get stale), use a simple approach: scan all deques to find the one with entries.
   
   - `pop_latest_undo(self) -> Optional[UndoEntry]` -- under lock, iterate all `_undo` deques, find any non-empty one, pop from it, push to corresponding `_redo` deque, return entry. Returns None if all empty. This is O(number of files) which is always small (typically < 10 files in a KiCad project).
   
   - `pop_latest_redo(self) -> Optional[UndoEntry]` -- same pattern for `_redo` deques.
   
   **L-03 note:** This approach eliminates the stale-reference problem entirely. No separate tracking fields needed. The "latest" is simply "whatever deque has entries."

```python
def undo(self, target_file: Optional[str] = None) -> dict[str, Any]:
    """Undo the most recent mutation for a file."""
    if self._undo_stack is None:
        return {"success": False, "error": "Undo stack not enabled"}

    if target_file is not None:
        file_path = (self._base_dir / target_file).resolve()
        entry = self._undo_stack.pop_undo(file_path)
    else:
        entry = self._undo_stack.pop_latest_undo()

    if entry is None:
        return {"success": False, "error": "No operations to undo"}

    # H-04: Symlink protection (mirrors Transaction H-02 control)
    if entry.file_path.is_symlink():
        return {"success": False, "error": "Security: target file is a symlink"}

    # M-08: Check parent directory exists before writing
    if not entry.file_path.parent.exists():
        return {"success": False, "error": "Cannot undo: parent directory no longer exists"}

    # L-05: Warn if file was modified externally since snapshot
    if entry.post_mtime and entry.file_path.exists():
        current_mtime = entry.file_path.stat().st_mtime_ns
        if current_mtime != entry.post_mtime:
            # File was modified externally -- still undo but warn
            pass  # Log warning if logging is available

    # L-04: Use newline="" to preserve exact byte content (LF line endings)
    entry.file_path.write_text(entry.pre_content, encoding="utf-8", newline="")

    # Invalidate cache for this file
    if self._cache:
        self._cache.invalidate(entry.file_path)

    return {
        "success": True,
        "undone_op": entry.op_type,
        "target_file": str(entry.file_path.relative_to(self._base_dir)),
    }
```

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| In-memory undo stack -> file system | undo/redo writes content from memory to disk (no external input) |
| Concurrent threads -> undo stack | Multiple threads may push/pop concurrently (mitigated by Lock) |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-33-01 | Denial of Service | UndoStack | mitigate | Bounded deque(maxlen=max_size) prevents unbounded memory growth. Default 50, configurable via KICAD_UNDO_MAX_SIZE. Document memory usage guidelines for large projects (M-07). |
| T-33-02 | Tampering | undo()/redo() | mitigate | File path resolved and validated against project directory before write (existing path confinement in executor) |
| T-33-03 | Tampering | Stale snapshots | accept+warn | If file modified externally after snapshot captured, undo restores stale content. Store post_mtime in UndoEntry and warn on mismatch (L-05). Same risk as any editor undo. |
| T-33-04 | Information Disclosure | UndoStack memory | accept | Undo entries stored in process memory only, not persisted. No exposure surface beyond process memory. |
| T-33-05 | Tampering | Symlink bypass in undo/redo | mitigate | Add is_symlink() check before writing in undo()/redo() (H-04). Mirrors Transaction H-02 control. |
| T-33-06 | Tampering | Concurrent write during undo | accept | No .lck file check -- undo is a fast operation, collision window is tiny. MCP context is single-session. Accept risk. |

## Known Limitations

| Limitation | Impact | Mitigation |
|------------|--------|------------|
| Multi-file undo is best-effort (M-03) | Cross-file undo may leave project inconsistent if one file write fails | Document in API response. Future: wrap in Transaction-style backup. |
| Session-scoped only | Undo history lost on MCP server restart | Standard editor behavior (KiCad also loses undo on close) |
| Create operations not undoable | Cannot undo file creation | Document in MCP tool description |
</threat_model>
