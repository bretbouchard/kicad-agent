    The CorpusCurator CLI pattern is:
    ```python
    from volta.training.corpus_curator import CorpusCurator
    from pathlib import Path
    import hashlib
    import json
    import random

    # Use existing fixtures as seed
    fixture_dirs = [
        Path("tests/fixtures"),
        Path("volta-0.1.0/tests/fixtures")
    ]

    # Collect existing .kicad_sch files
    schematics = []
    for fdir in fixture_dirs:
        if fdir.exists():
            for sch in fdir.rglob("*.kicad_sch"):
                sha256 = hashlib.sha256(sch.read_bytes()).hexdigest()
                schematics.append({"path": str(sch), "sha256": sha256})

    # If < 100, download more via CorpusCurator
    if len(schematics) < 100:
        curator = CorpusCurator()
        projects = curator.curate_batch()
        # Copy schematics from temp dirs to corpus directory
        # The curate_batch() downloads to tempdir and returns CuratedProject objects

    # Sample with seed=42 for reproducibility
    random.seed(42)
    random.shuffle(schematics)
    selected = schematics[:1000]  # Target 1000 or what's available

    # Create manifest.json
    manifest = {
        "count": len(selected),
        "seed": 42,
        "schematics": selected
    }
    with open("corpus/manifest.json", "w") as f:
        json.dump(manifest, f, indent=2)
    ```

## Task 2: Implement Python parity driver

<task>
  <id>234a-02</id>
  <title>Implement Python Parity Driver</title>
  <files>
    <create>.planning/phases/234a-corpus-and-driver/scripts/batch_erc_parity.py</create>
  </files>
  <action>
    Create `scripts/batch_erc_parity.py` with:

    1. `run_python_erc(schematic_path)` - invokes `run_native_erc(Path(...))` from `volta.validation.native_erc`
    2. `run_swift_erc(schematic_path)` - invokes Swift NativeERC via subprocess using xcrun simctl or direct binary. The macOS app has a SwiftUI interface but we need a CLI entry point. Use the pattern from ValidationPanel.swift: call the macOS binary with --erc-batch flag or create a test harness.
    3. `normalize_result(result)` - converts Python or Swift result to common dict format with keys: clean, error_count, warning_count, violations (list of dicts with check_id, severity, ref, net)
    4. `compare_results(python_result, swift_result)` - returns dict with agreement, fp (Swift finds what Python misses), fn (Python finds what Swift misses)
    5. `parity-test` subcommand - runs ONE schematic and prints both Python and Swift results side by side

    For Swift invocation, since there's no existing CLI, the approach is to build and run the macOS app in test mode, or use the Python script to invoke via NSApplication on macOS. For Phase 234A, we create the driver structure that 234B will use. The smoke test will verify the structure works.

    The Swift app's NativeERC.swift is a static method that can be called from XCTest or via a Swift script. For the parity driver, we'll use subprocess to run swift command or build a test harness.

    Key insight: The macOS app is built separately. For development, we can use `swift test` with a test case that imports the module, or create a macOS CLI target. For Phase 234A, create the driver that will work once the macOS app is built.

    For now, implement Python side fully and create Swift invocation placeholder that will be completed in Phase 234B.
