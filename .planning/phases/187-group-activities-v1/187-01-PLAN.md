  <name>Task 1: Define GroupActivities session lifecycle</name>
  <files>KiCadAgent/Collaboration/GroupActivitiesManager.swift, KiCadAgent/Collaboration/EventSync.swift</files>
  <action>
Create GroupActivities session manager following ARCHITECTURE.md Pattern 6:

**KiCadDesignSession GroupActivity:**
- struct KiCadDesignSession: GroupActivity
- metadata: title = "KiCad Design Session", type = .generic
- DecisionEvent payload: conversationId, eventId, eventType, payload (Data), timestamp, author
- Minimal delta sync: event payloads only, no full KiCad files (files sync via iCloud Drive separately per LIVE-05)

**GroupActivitiesManager:**
- startSession() async throws → creates KiCadDesignSession, calls join()
- joinSession(token: Data) async throws → joins existing session via FaceTime link
- messenger: GroupSessionMessenger for message passing
- subscribeToEvents() async → for await loop over messenger.events
- broadcastDecision(_ decision: Decision) async throws → serialize to DecisionEvent, send via messenger
- broadcastCursor(position: CGPoint, selection: UUID?) async throws → cursor position sync (LIVE-03, defer to Phase 189)
- participantStates: [UUID: ParticipantState] tracking presence, join time

**Event synchronization:**
- handleIncomingEvent(_ message: GroupSessionMessage) async → deserialize DecisionEvent
- Apply to local SwiftData: insert Decision/Message/ValueChange
- Trigger UI update: @Published events feed, decision timeline refresh
- Versioning: each DecisionEvent has version number to detect out-of-order arrival
- Replay on reconnect: missed events fetched from CloudKit journal by version number

**Participant cap (LIVE-02):**
- checkParticipantCount() before join: if >=4, throw SessionError.sessionFull
- Session full UI: "Session full (4/4)" banner with "Notify when slot opens" button
- Notification queue: when participant leaves, notify queued 5th participant via local notification
- Cap enforced server-side by GroupActivities framework

**Network drop recovery (LIVE-06):**
- Auto-reconnect: on messenger disconnect, retry with exponential backoff (1s, 2s, 4s, 8s, max 30s)
- Replay missed events: fetch from CloudKit by timestamp range (disconnectedAt → reconnectedAt)
- Event versioning: track lastProcessedVersion, request replay from that version
- Graceful degradation: show "Reconnecting..." state, buffer local events until reconnected

<task type="auto">
  <name>Task 2: Implement auto-handoff on initiator leave</name>
  <files>KiCadAgent/Collaboration/SessionHandoffManager.swift</files>
  <action>
Create SessionHandoffManager for clean session termination:

**Initiator leave detection (LIVE-07):**
- Track initiator UUID (session.creator from GroupActivities metadata)
- Subscribe to participantStateChanged events
- On initiator .left: trigger auto-handoff

**Auto-handoff logic:**
- Find longest-present participant (max(joinTime) among active participants)
- If all left simultaneously (rare): session ends naturally
- Promote new initiator: call handoffInitiator(to: participant) on session
- Show banner to all participants: "Initiator left. [New Name] is now session host."

**Manual handoff:**
- Initiator can explicitly handoff: "Make [Name] host" action
- Prompts for confirmation (irreversible action guard)
- Transfers sessionInitiator role via GroupActivities API

**Session cleanup on end:**
- Unsubscribe from messenger.events
- Clear participantStates cache
- Persist final event list to SwiftData (session summary)
- Notify participants: "Session ended. Decisions saved to project."

**Edge cases:**
- Initiator abrupt leave (app crash, network loss): auto-handoff still triggers
- Single participant left: session ends, no handoff needed
- All participants leave within 5 seconds: session ends, summary saved

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Session Participant → Event Sync | Untrusted decision payloads, cursor positions |
| GroupActivities Messenger | Network-reliable messaging, but out-of-order delivery possible |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-187-01 | S | EventSync | mitigate | Validate DecisionEvent schema before SwiftData insert, reject malformed payloads |
| T-187-02 | T | GroupActivities messenger | accept | GroupActivities framework encrypts in transit, Apple handles security |
| T-187-03 | I | Participant cap | mitigate | Server-side cap enforcement, client-side check before join |
| T-187-04 | D | Auto-handoff | accept | Handoff is role transfer, no data loss, requires active participant |
| T-187-05 | E | Reconnect flood | mitigate | Exponential backoff max 30s, max 5 reconnect attempts before manual retry required |

## Adversarial Scenario

Name: Decision Event Injection Attack
Malicious participant crafts invalid DecisionEvent payload (wrong eventType, corrupted data) to corrupt local SwiftData or crash the app.

Mitigation:
1. Validate DecisionEvent schema: eventType must be in allowed set, payload size <1MB
2. Reject malformed events: drop silently, log to audit trail
3. Versioning: track lastProcessedVersion, skip dupes/out-of-order
4. SwiftData transaction rollback on insert failure
</threat_model>
