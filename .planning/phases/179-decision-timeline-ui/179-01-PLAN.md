<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Timeline query | SwiftData fetches potentially large datasets — must paginate |
| LLM chapter suggestion | External model generates chapter boundaries — validate output |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-179-01 | T (Tampering) | Snapshot preview images | mitigate | Cache hash validated on load. Reject corrupted images. |
| T-179-02 | I (Information disclosure) | Timeline search query | accept | Search filters local events. No external exposure. |
| T-179-03 | D (Denial of Service) | Pagination without limit | mitigate | Hard cap 100 events/page. Max 1000 pages (100K events). Beyond → error alert. |
| T-179-04 | E (Elevation) | LLM chapter suggestion | mitigate | Parse LLM response as Chapter structs, validate indices (0 ≤ start < end ≤ eventCount). Reject invalid ranges. |
| T-179-05 | R (Repudiation) | Snapshot restore log | mitigate | Restore Decision auto-logged with timestamp + actor. |

## Adversarial Scenario

**Name:** Timeline query without pagination causes OOM on 100K events

**Attack:** Malicious conversation with 1M Decision events. DecisionTimelineView fetches all events without pagination. App OOM crashes, loses unsaved work.

**Mitigation:** Hard pagination cap: `FetchDescriptor.fetchLimit = 100`. Max pages 1000 (100K events). Beyond → alert: "Timeline limit reached (100K events). Use filters to narrow scope." Test `testTimelinePaginationCap` enforces limit per Pitfall 8.

**Detection:** Unit test mocks 101K events, expects timeline to stop at 1000th page with error alert.
</threat_model>
