     - **Step 1:** Run SPICE validation (from Task 1)
       - Call `SPICEValidator.validate(schematic_path)`
       - Get SPICEResult

   - On validation failure:
     - **T1 (first failure):** Retry with relaxed constraints
       - SPICE: Relax convergence tolerance (GMIN stepping)
       - ERC: Treat warnings as non-blocking
       - DRC: Reduce clearance requirements
     - **T2 (second failure):** Switch strategy
       - Request user to fix circuit constraints (remove short, add ground)
       - Ask user to simplify circuit (reduce complexity)
     - **T3 (third failure):** Request human input
       - Surface failure as user decision: "Generation failed: {error_details}. Please fix conversation or constraints."
       - Require user acknowledgment before continuing
     - **T4 (fourth failure):** Halt generative pipeline
       - Log to OpJournal: "Generation halted after 4 failures: {failure_summary}"
       - Return CorrectnessResult(valid=False, failures=all_errors)

3. **Escalation state tracking**:
   - Track failure counts per validation type (SPICE, ERC, DRC)
   - Store in OpJournal (per Phase 169 pattern)
   - Reset on successful validation

   - All failures raised as StructuredError with context:
     - error_type: "SPICE_CONVERGENCE_FAILED" / "ERC_UNCONNECTED_PIN" / "DRC_CLEARANCE_VIOLATION"

     - details: Structured error details (component ref, pin number, violation type)
     - recovery_hint: Actionable suggestion ("Add wire from R1:1 to net GND")
   - Per GENOUT-08: Escalation surfaces failures as user decisions (not silent failures)

## Adversarial Scenario

**Name:** "SPICE netlist injection attack via malicious schematic"

**Scenario:** Attacker with conversation access injects malicious SPICE directives:
- User prompt includes SPICE directives in conversation ("include malicious.lib")
- SKIDL compiler includes directives in generated schematic
- Netlist converter exports .include directive in SPICE netlist
- ngspice processes malicious.lib (exfiltrates circuit data via .alter)

**Mitigation:**
1. Validate SPICE netlist before ngspice execution:
   - Reject .include directives (only allow built-in device models)
   - Reject .alter directives (prevent parameter exfiltration)
   - Reject shell escapes (no ! directives)
2. Sanitize schematic before netlist conversion (remove SPICE comments, whitelist only KiCad fields)
3. Run ngspice in subprocess with timeout (30s), kill if hangs
4. Parse ngspice output for suspicious activity (file access, network connections)

Per GENOUT-08: SPICE validation ensures correctness. Netlist injection breaks correctness (malicious circuits fail validation). Sanitization prevents injection.
</threat_model>
