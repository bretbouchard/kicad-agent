<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| App → KiCad CLI | Semi-trusted subprocess (GPL tool, sandboxed) |
| HTTP MCP (opt-in) → Daemon | Untrusted external clients (localhost only, auth token) |
| User → Settings UI | Trusted local interaction (system preferences) |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-163-01 | S | KiCad CLI invocation (subprocess) | mitigate | Validate kicad-cli path before execution, sandbox restricts to user directories, reject absolute paths outside project. |
| T-163-02 | I | KiCad version detection | accept | Version string from official KiCad binary, trusted source. No injection possible. |
| T-163-03 | D | KiCad file operations | mitigate | App Sandbox entitlements, validate file paths in daemon ops, kicad-cli respects sandbox. |
| T-163-04 | E | HTTP MCP auth token exposure | mitigate | UUID v4 token, localhost-only binding (127.0.0.1), rate limiting (10 req/s), auto-ban after 100 failed auth per DAEM-08. |
| T-163-05 | R | HTTP MCP privilege escalation | mitigate | HTTP server runs with same privileges as app, no sudo/root, auth token required for all ops. |
| T-163-06 | T | HTTP MCP token theft (local) | mitigate | Localhost-only binding, no network exposure, token rotation available, audit log tracks connections. |

## Adversarial Scenario

**Attack:** Attacker exploits HTTP MCP auth token (leaked via log dump) to execute malicious KiCad operations. **Mitigation:** (E) Token auto-revokes after 100 failed auth per DAEM-08, (R) Rate limiting prevents brute force, (T) Localhost-only binding prevents network exposure, (D) Sandbox restricts file damage, (I) App requests user approval before destructive ops (approval gates).
</threat_model>
