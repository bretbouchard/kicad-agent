## Objective

Implement three read-only ERC infrastructure operations: `parse_erc` (structured ERC report parsing), `extract_violation_positions` (filter ERC violations by type and extract positions), and `validate_hlabels` (verify hierarchical labels match expected set). These are foundational tools that Plan 02-04 depend on for ERC-driven repair workflows.

Purpose: These read-only operations provide the structured data access layer that all mutation operations in subsequent plans consume. Without them, the repair operations cannot determine what to fix or where.

Output: Two new modules (`erc_parser.py`, `hlabel_guard.py`), three new Pydantic op models in `schema.py`, three handler registrations in `executor.py`, and comprehensive test coverage.

2. **`validate_hlabels(ir: SchematicIR, expected_labels: set[str] | None = None) -> HlabelValidationResult`**:
   - Access `ir.schematic.hierarchicalLabels` to get the actual set of hlabel text strings.
   - If `expected_labels` is provided (not None): compare actual vs expected. Missing = expected - actual, Extra = actual - expected. `passed = len(missing) == 0`.

## Threat Model

|----------|-------------|
| LLM intent -> parse_erc | Untrusted input: violation_type string must be validated by Pydantic (min_length, max_length, safe_identifier pattern) |
| kicad-cli output -> parser | External tool output: JSON parsing must handle malformed data gracefully |

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-23-01 | T | violation_type field | mitigate | Pydantic min_length/max_length validation + safe_identifier check prevents injection |
| T-23-02 | D | kicad-cli output | accept | JSON parse errors caught and returned as structured error results (not exceptions) |
| T-23-03 | T | expected_labels list | mitigate | Pydantic max_length=200 prevents excessive label lists; individual strings max_length=128 |
