<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| CI Environment → Codebase | CI runs untrusted code from PRs (tampering risks) |
| Coverage Reports → Enforcement | Tampered coverage reports could hide gaps (integrity risk) |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-200-01 | T | False coverage report (tampering) | mitigate | Generate report in CI (no PR tampering), checksum verification |
| T-200-02 | T | False positive coverage (mocking) | mitigate | 100% line + branch required (harder to fake), human review suspicious 100% |
| T-200-03 | R | CI workflow bypassed | mitigate | Branch protection requires quality-gate.yml pass before merge |
| T-200-04 | I | Mutation score tampering | mitigate | Score calculated from raw mull/mutmut output in CI (no PR influence) |
| T-200-05 | D | ThreadSanitizer report deleted | mitigate | Upload report as artifact (retained 30 days), deletion doesn't erase failure |

## Adversarial Scenario

Name: Attacker submits PR with tampered coverage report. Attacker modifies scripts/enforce_coverage.py to always return 100% coverage without actually running tests. PR claims "100% coverage added" but actually adds untested code. Reviewer checks coverage report (shows 100%), approves PR.

Mitigation: CI workflow runs tests in fresh environment (attacker cannot influence). Coverage report generated by xcrun llvm-cov / pytest-cov (not attacker-provided). enforce_coverage.py reads fresh report, calculates percentage from raw data (no tampering). Branch protection requires CI pass before merge. Attacker cannot bypass CI without breaking build.
</threat_model>
