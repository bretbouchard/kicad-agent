Create OpJournal per ARCHITECTURE.md Pattern 7:

1. **OpJournal.swift**:
   - `struct OpJournal { let fileURL: URL; let queue: DispatchQueue; let projectID: UUID }`
   - `func append(_ entry: OpJournalEntry) throws`: 
     - Encode entry to JSON line (ISO8601Date, operationType, operationId, parameters [String: AnyCodable], result [String: AnyCodable], executionTimeMs, phase).
     - Append line to file with trailing `\n`.
     - `DispatchQueue.async` for non-blocking writes.
     - `FileHandle.seekToEndOfFile`, `handle.write(line.data)`, `handle.synchronizeFile()` for fsync durability (critical per routing/audit.py H5).
   - `func queryByOperation(_ operationId: UUID) throws -> [OpJournalEntry]`: Read file, filter by operationId, return matches.
   - `func queryByPhase(_ phase: GSDPhase) throws -> [OpJournalEntry]`: Filter by phase field.
   - **Partitioning per project**: OpJournal.fileURL = `.libraryDirectory.appendingPathComponent("op_journals/<projectID>.jsonl")`. Each project has separate journal file (query pruning).

2. **OpJournalEntry struct** (Codable):
   ```swift
   struct OpJournalEntry: Codable {
     var timestamp: ISO8601Date
     var operationType: String
     var operationId: UUID
     var parameters: [String: AnyCodable]
     var result: [String: AnyCodable]
     var executionTimeMs: Int64
     var phase: GSDPhase
     var requirementId: String?  // GOV-11 linkage
   }
   ```

3. **Integration with Decision events**: Every op append triggers Decision insert: `Decision(eventType: .opCall, eventData: encoded(entry), reasoning: "Executed \(operationType)")`.

**Append-only enforcement**: No delete methods. Journal grows monotonically. Compaction (Task 2) archives old files but never mutates existing lines.

Create EventCompactor per Pitfall 8 prevention:

1. **EventCompactor.swift**:
   - `struct EventCompactor { let context: ModelContext; let journal: OpJournal }`
   - `func compactOldEvents(for project: Project) async throws`:
     - Fetch all Decision/ValueChange/Message entities for project.
     - If count > 100K, archive oldest 50K to separate store.
     - **Archive strategy**: 
       - Create archive JSONL: `project.archive/<timestamp>_events.jsonl` (flat file in iCloud Drive).
       - Serialize archived entities to JSON lines, write with fsync.
       - Delete archived entities from SwiftData: `context.delete(oldMessages)`, `context.delete(oldDecisions)`.
       - Create checkpoint snapshot (ProjectSnapshot) at archive point to enable replay.
     - Keep last 50K events active (most recent).
   - `func restoreFromArchive(_ archiveURL: URL) async throws`: 
     - Read archive JSONL, decode entities, insert into SwiftData.
     - Replay events up to checkpoint snapshot.

2. **Materialized snapshots per requirement**: Maintain `ProjectSnapshot` every 100 events (auto-snapshot per MEM-07). Query for current state uses snapshot, not event replay. Snapshots stored in SwiftData (not archived).

3. **Active event count**: `func activeEventCount(for project: Project) -> Int`: Query SwiftData, return count. Trigger compaction if > 100K.

**Compaction schedule**: Run compaction on app background (15 min after last event edit). User can trigger manual compaction via Settings → "Archive old events".

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| OpJournal file | Local JSONL with fsync — corrupt file loses audit trail |
| Archive restore | User restores from archive — must validate integrity |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-180-01 | T (Tampering) | OpJournal JSONL | mitigate | SHA256 hash written after each line. Validate on read. Reject corrupted files. |
| T-180-02 | I (Information disclosure) | Archive files | accept | Archives contain conversation history (no secrets). Stored in iCloud Drive (encrypted). |
| T-180-03 | D (Denial of Service) | Compaction without limit | mitigate | Min 50K events always active. Max compaction 1 per day. Prevents churn. |
| T-180-04 | E (Elevation) | Restore from archive | mitigate | Validate archive SHA256 hash on restore. Reject corrupted archives. |
| T-180-05 | R (Repudiation) | OpJournal signature | mitigate | Entries signed with timestamp + actor. Cannot forge. |

## Adversarial Scenario

**Name:** OpJournal file corruption causes data loss, compaction silences error

**Attack:** Malicious actor corrupts OpJournal JSONL (inject invalid JSON). OpJournal.read() crashes, user loses audit trail. Compaction runs, archives corrupt events, user cannot recover.

**Mitigation:** OpJournal.read() validates JSON per line. Invalid lines logged but skipped. File hash (SHA256) stored in separate metadata file. On mismatch, alert user: "OpJournal corrupted, restore from backup?". Test `testCorruptedOpJournalRecoverable` validates recovery path.

**Detection**: Unit test injects invalid JSON line, expects read() to skip invalid entry, log warning, return valid entries.
</threat_model>
