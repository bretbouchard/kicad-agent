Create LLMProvider.swift protocol to abstract all AI providers:
```swift
protocol LLMProvider {
    associatedtype Config: Sendable
    associatedtype Message: Sendable

    var providerID: String { get }
    var modelID: String { get }
    var isAvailable: Bool { get }

    init(config: Config) throws

    func generateText(
        prompt: String,
        maxTokens: Int?,
        temperature: Double?
    ) async throws -> String

    func generateVision(
        prompt: String,
        image: Data
    ) async throws -> String  // For Gemma 4 vision tasks

    func estimateCost(
        inputTokens: Int,
        outputTokens: Int
    ) -> Decimal  // Returns 0.0 for local models
}
```

Key design decisions:
- Associated types for provider-specific config/messages (SDK types don't leak)
- `isAvailable` for runtime availability checks (FoundationModels on Intel Macs)
- `estimateCost` returns 0.0 for local models (FoundationModels, MLX-Swift)
- `generateVision` separate method for image inputs (PCB analysis, routing)

Per MOD-01: App uses KiCadModelProvider protocol as only model interface (SDK types don't leak).

Per STACK.md Swift AI SDK pattern: Unify 37+ providers via one protocol, swap providers without changing call sites.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| App → FoundationModels | Trusted local framework (Apple-signed, no network) |
| App → MLX-Swift | Trusted local framework (Metal, no network after download) |
| App → HF Hub | Semi-trusted HTTPS (model downloads, SHA256 verification) |
| User → Model Catalog | Trusted UI (user-initiated downloads) |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-164-01 | S | HF Hub model download (supply chain) | mitigate | SHA256 verification against known-good manifest, reject mismatched models per STACK.md security. |
| T-164-02 | I | FoundationModels availability check | accept | Apple framework, trusted source. No injection possible. |
| T-164-03 | D | MLX model file execution | mitigate | Validate .mlx format before loading, reject non-MLX files per MOD-08, sandbox restricts file access. |
| T-164-04 | E | HF Hub download failure | mitigate | Auto-resume on app relaunch per MOD-07, cache partial downloads, retry with exponential backoff. |
| T-164-05 | R | MLX model privilege escalation | accept | MLX-Swift runs in app process, no extra privileges, Metal access restricted to app sandbox. |
| T-164-06 | T | FoundationModels telemetry (Apple) | accept | Apple Intelligence may use anonymized telemetry for improvements, per Apple privacy policy. |

## Adversarial Scenario

**Attack:** Attacker supplies poisoned model via HF Hub download (supply chain attack). **Mitigation:** (S) SHA256 verification rejects modified .safetensors per STACK.md, (D) MLX format validation rejects non-MLX files, (E) Download failure auto-resumes per MOD-07, (I) FoundationModels unavailable gracefully degrades to MLX-Swift, (T) Apple telemetry not applicable to MLX-Swift (local-only).
</threat_model>
