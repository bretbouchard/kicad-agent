<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| XCUITest → App UI | XCUITest drives UI via accessibility APIs, untrusted input simulates user actions |
| Simulator/Test Device | Test environment may have different permissions/sandbox than production |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-196-01 | S | XCUITest target | accept | XCUITest requires same entitlements as app (safe, same developer team) |
| T-196-02 | T | Time-travel slider | mitigate | Validate slider bounds within event index range (prevent index out-of-bounds) |
| T-196-03 | I | Collaboration invite test | mitigate | Test uses real CKShare — ensure test data doesn't leak to production users (use test-only CloudKit container) |
| T-196-04 | D | Snapshot restore test | accept | Test fixture data only, no production data at risk |
| T-196-05 | E | XCUITest flakiness | mitigate | Use stable accessibilityIdentifiers, waitForExistence with timeout, no UUID-based queries |

## Adversarial Scenario

Name: Malicious test fixture injects poisoned decision events into SwiftData during XCUITest. An attacker crafts a test fixture with decision events containing SQL injection payloads in event payloads. When XCUITest loads fixture and SwiftData persists events, CloudKit sync propagates malicious data to user's production iCloud account.

Mitigation: XCUITest uses test-only CloudKit container (CKContainer.identifier() returns "com.kicadagent.test" during test runs, separate from production "com.kicadagent.projects"). Test fixtures signed with test-only keychain access. Production container never accessed during test execution.
</threat_model>
