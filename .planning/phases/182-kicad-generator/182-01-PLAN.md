   - `run_erc(schematic_path: Path) -> ERCResult`:
     - Run `kicad-cli sch erc` via subprocess (per CLAUDE.md tool inventory)

     - Return ERCResult(valid=returncode==0, errors=error_list)
     - Critical errors: unconnected pins, duplicate refs, power conflicts
     - Warnings: deprecated symbols, off-grid components (log but don't fail)

   - `run_drc(pcb_path: Path) -> DRCResult`:
     - Run `kicad-cli pcb drc` via subprocess
     - Parse DRC output (check return code, stderr for errors)
     - Return DRCResult(valid=returncode==0, errors=error_list)
     - Critical errors: clearance violations, missing connections, drill errors

3. **Obdurate escalation integration** (per GOV requirements):

## Adversarial Scenario

**Name:** "kicad-cli symlink attack via PATH manipulation"

**Scenario:** Attacker with file system access replaces `kicad-cli` with malicious symlink or script:
- Symlinks `kicad-cli` to malicious script that exfiltrates circuit data
- Script exits with 0 (passes ERC/DRC despite corrupt files)
- Generated KiCad files contain backdoors or corrupted data

**Mitigation:**
1. Validate kicad-cli path before execution:
   - Check absolute path (`/usr/local/bin/kicad-cli` or user-specified)
   - Verify file is executable and signed (macOS code signature check)
   - Reject symlinks: `os.path.realpath() == os.path.abspath()`
2. Hash kicad-cli binary on first use, detect changes
3. Run kicad-cli in subprocess with timeout (30s), kill if hangs
4. Parse ERC/DRC output for expected format (malicious tool won't produce valid KiCad output)

Per GOV requirements: External tool failures trigger Obdurate escalation. kicad-cli not found → T1: Prompt user to install KiCad.
</threat_model>
