<objective>
Fix two position-calculation bugs that actively make schematics worse:

**R-2 (P0-002):** `place_missing_units` places multi-unit components at colliding positions. The dedup set (`_occupied_positions`) exists but only runs in the fallback path — when `_find_position_for_unit` returns a position, it bypasses the collision check. Fix: route ALL returned positions through the dedup set.

**R-4 (P0-004):** `place_no_connects_from_erc` places markers at wrong positions because `pos_to_type` dict uses `round(x, 2)` keys (10μm grid) while ERC violation coordinates have sub-micron precision. Fix: replace exact dict key lookup with tolerance-based matching using the existing `SNAP_TOLERANCE`.

Per CONTEXT.md D-3: General fixes only, no backplane-specific workarounds.
Per CONTEXT.md D-6 (Claude's Discretion): Reuse existing position-transform patterns from `schematic_ir.py:get_pin_positions()`.

Purpose: These two bugs together create +31 new ERC violations per affected component on the backplane. Both share a theme of position-matching inconsistency but have distinct mechanisms.

Output: Two targeted fixes in repair_components.py and repair_erc.py, plus regression tests for both.
</objective>

   def _lookup_pin_type_with_tolerance(
       x: float,
       y: float,
       pin_positions: list[dict],
       tolerance: float,
   ) -> str:
       """Find pin electrical type at (x,y) within tolerance. Default 'passive'.

       Replaces exact dict-key lookup which fails when ERC violation positions
       and pin positions differ by sub-micron precision (round to different
       2-decimal keys). Uses the same SNAP_TOLERANCE pattern as _near_anchor.
