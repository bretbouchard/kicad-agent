Create SwiftUI collaboration view following ARCHITECTURE.md collaboration patterns:

**Full project context (COLLAB-05):**
- @Query Conversation: loads full conversation history
- @Query Decision: loads all decisions with timestamps
- @Query ValueChange: loads all value changes
- Render views: inline schematic SVG, PCB PNG (from Phase 172)
- All collaborators see same content (CloudKit sync)

**Permission-aware actions (COLLAB-06):**
- Check CKShare.userRole and participantPermission before enabling UI
- View-only: disable "Send message" button, disable gate approval buttons
- Edit permission: enable all actions
- Fork permission: enable "Fork project" button (creates new project)
- Show permission badge: "View only" / "Can edit" / "Can fork"

**Local daemon execution (COLLAB-07):**
- User action → local Python daemon executes op (stdio MCP from Phase 167)
- Daemon runs on collaborator's Mac (their subprocess, their KiCad install)
- Result stored to SwiftData (Decision, ValueChange, OpJournalEntry)
- CloudKit sync propagates to other devices
- No file upload/download (KiCad files not synced per LIVE-05)

**Gate approval UI:**
- Approval gates (ERC warnings, op confirmation) surface for editors
- View-only collaborators see gates as read-only: "Awaiting [Editor Name] approval"
- Editors see approve/reject/show-me buttons
- Approval decisions journaled as Decision events

**Conversation history rendering:**
- Message list: user messages + assistant responses
- Inline renders: schematic previews (SVG), PCB renders (PNG)
- Cost badges: token usage, model used
- Decision links: tap decision → jump to Decision Timeline (Phase 179)

**Derived file regeneration (LIVE-05):**
- Each Mac regenerates .kicad_sch/.kicad_pcb locally from conversation state
- No iCloud Drive file sync (files stored in local .kicadagent bundle)
- CloudKit syncs conversation events, not KiCad files
- Prevents file conflicts (Pitfall 6 prevention)

**Error handling:**
- Daemon not running: show "Daemon unavailable. Restart daemon?" prompt
- CloudKit sync paused: show "Sync paused. Waiting for connection." banner
- Permission changed: force UI reload to apply new ACL

  <name>Task 3: Implement LWW conflict resolution with prompts</name>
  <files>Volta/Collaboration/ConflictResolver.swift</files>
  <action>
Create ConflictResolver for LWW conflict resolution (COLLAB-10, MEM-10):

**Conflict detection:**
- On ValueChange sync from CloudKit
- Compare local ValueChange.timestamp vs remote ValueChange.timestamp
- If timestamps differ (both edited same value within sync window): conflict detected
- Conflict window: 5 seconds (tunable per MEM-10)

**LWW resolution (MEM-10 pattern):**
- Last-Writer-Wins based on timestamp
- If remote.timestamp > local.timestamp: remote wins (prompt user about incoming change)
- If local.timestamp > remote.timestamp: local wins (user already has latest)

**Prompt UI (COLLAB-10):**
- Show alert: "Collaborator updated [Value Name]"
- Side-by-side diff:
  - Left: "Your version" (local value)
  - Right: "Their version" (remote value)
- Actions: "Keep yours", "Keep theirs", "Merge" (if mergeable)
- If user picks "Keep theirs": update local ValueChange, trigger UI refresh
- If user picks "Keep yours": dismiss prompt, no change
- Merge: manual merge UI (deferred to future phase, auto-pick theirs for v1)

**Conflict logging:**
- All conflicts logged to OpJournal
- Conflict type, resolution chosen, actor, timestamp
- Audit trail for debugging sync issues

**Edge cases:**
- Simultaneous edit on different values: no conflict, both apply
- Simultaneous edit on same value: conflict, prompt user
- 3+ participants editing same value: LWW applies, prompt user about latest change only

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| CloudKit Sync → Local State | Remote ValueChanges may conflict with local edits |
| Permission Checks → UI Actions | Untrusted permission metadata from CKShare |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-189-01 | S | ValueChange sync | mitigate | Validate ValueChange schema, reject malformed payloads, sanitize user input |
| T-189-02 | T | Permission checks | mitigate | Client-side check before op (Pitfall 10), verify CKShare metadata on every action |
| T-189-03 | I | Conflict resolution | accept | LWW with prompts, user chooses winner, audit log prevents silent overwrites |
| T-189-04 | D | Activity feed flooding | mitigate | Rate limit events per participant (max 100/sec), paginated query |
| T-189-05 | E | Permission bypass | mitigate | All permission checks in SwiftData transaction, rollback on violation |

## Adversarial Scenario

Name: Permission Elevation via CKShare Metadata Spoofing
Malicious collaborator crafts CKShare metadata claiming .readWrite permission while server-side record shows .readOnly.

Mitigation:
1. Client-side check reads CKShare.userRole from CloudKit server truth
2. Server-side CKShare enforces permission on write operations
3. Audit log: all permission changes logged, detect drift
4. Fallback: if client check fails, disable edit UI, show "Permissions unavailable. Refresh?" prompt
</threat_model>
