# Phase 106: Vast.ai Training Run Guide

**Goal:** Repoint the Gemma 4 12B vision model to include blocker-diagnosis
capability (Phase 104) alongside its existing spatial reasoning skills.

**Approach:** SFT on the merged dataset (142,752 rows = 142K existing + 110
new diagnostic). Start with SFT, defer GRPO (council Q4 guidance).

## Prerequisites

1. **Vast.ai account** with a GPU instance (RTX 4090 or A100 recommended).
2. **HuggingFace token** with access to `google/gemma-4-12b-it`.
3. **Dataset uploaded** to the instance (see Packaging below).

## Packaging the training bundle

The following files need to be on the Vast.ai instance:

```bash
# On your local machine:
tar czf phase106_training_bundle.tar.gz \
    scripts/vast_train_kicad.py \
    training_output/phase106_dataset/ \
    output/kicad_vision_adapter_v2/checkpoint-2000/adapter_config.json
```

Upload to Vast.ai:
```bash
scp phase106_training_bundle.tar.gz root@<vast-ip>:/workspace/
```

On the instance:
```bash
cd /workspace
tar xzf phase106_training_bundle.tar.gz
```

## Environment setup (on the instance)

```bash
pip install torch transformers peft trl datasets accelerate bitsandbytes
export HF_TOKEN=<your-huggingface-token>
```

## Training command

```bash
python3 /workspace/scripts/vast_train_kicad.py \
    --dataset_path /workspace/training_output/phase106_dataset/train \
    --output_dir /workspace/phase106_adapter \
    --model_id google/gemma-4-12b-it \
    --lora_rank 64 \
    --lora_alpha 128 \
    --max_steps 2000 \
    --learning_rate 1e-5 \
    --batch_size 1 \
    --gradient_accumulation_steps 8 \
    --max_seq_length 2048
```

**Notes on parameters:**
- `lora_rank 64` / `lora_alpha 128` — matches the prior adapter (checkpoint-2000).
- `max_steps 2000` — matches prior training. The 110 diagnostic rows are a
  small fraction of 142K, so the model won't overfit on them but will learn
  the task type.
- The prior run took ~5.1 hours on an A100 with these settings.

## Expected outcome

The adapter will learn to:
1. Continue performing its existing spatial reasoning tasks (maze routing,
   schematic analysis, board analysis) — these dominate the dataset.
2. Perform a new `blocker_diagnosis` task: given a routing failure
   description (net name, dead-end point, target point, failure type),
   identify the blocker and recommend an action.

## Post-training

1. Download the adapter from Vast.ai:
   ```bash
   scp -r root@<vast-ip>:/workspace/phase106_adapter/ output/phase106_adapter/
   ```

2. Convert to MLX format for local inference:
   ```bash
   python3 scripts/convert_peft_to_mlx.py \
       --peft-path output/phase106_adapter/ \
       --output output/phase106_adapter_mlx/
   ```

3. Verify the adapter loads and produces blocker diagnoses:
   ```bash
   python3 scripts/evaluate_gemma_adapter.py \
       --adapter output/phase106_adapter_mlx/ \
       --test-data training_output/diagnostic_sft_combined.jsonl
   ```

## Dataset details

**Merged dataset:** `training_output/phase106_dataset/train` (142,752 rows)

| Task type | Rows | Source |
|-----------|------|--------|
| maze_routing | 135,946 | Existing unified vision data |
| schematic_analysis | 2,902 | Existing |
| component_knowledge | 1,877 | Existing |
| board_analysis | 1,080 | Existing |
| domain_knowledge | 834 | Existing |
| **blocker_diagnosis** | **110** | **Phase 106 (NEW)** |
| routing_quality | 3 | Existing |

The diagnostic SFT data was generated by:
1. `scripts/generate_diagnostic_training_data.py` — injects keepout zones
   into Arduino_Mega, smd_test_board, and phase99_synthetic fixtures.
2. Phase 104 `BlockerDiagnostician` produces ground-truth labels.
3. `scripts/build_phase106_dataset.py` merges with existing data.

> harvested diagnostic traces** (cheap, deterministic labels from Phase 104).

decide whether GRPO (with real DRC reward) is warranted.
